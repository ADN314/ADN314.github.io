### 1. Structure of a Security Alert

A high-quality alert must provide enough data for an analyst to answer the **5W1H** questions (Who, What, When, Where, Why, How).

![image.png](Homework%20-%20Day%202%20(05%2001%202026)/image.png)

- **Alert Name / Rule ID:** A short, on-point name and the logic/condition that triggered it.
- **Source:** The origin of the activity (User, Host, or IP).
- **Destination:** The target of the activity (Service, Domain, or IP).
- **Timestamp:** When the event occurred and when the alert was triggered (note: there may be latency between these two).
- **Action:** Whether the system **Allowed**, **Blocked**, **Succeeded**, or **Failed** the activity.
- **Severity:** The level of seriousness (Critical, High, Medium, Low, or Informational).
- **Artifacts / IOCs:** Evidence such as URLs, file hashes, process names, or malicious IPs.
- **Context:** Additional info to determine impact, such as if the user is a **VIP** or if the asset is critical.

## 2. Alert Classification

![image.png](Homework%20-%20Day%202%20(05%2001%202026)/image%201.png)

| **Classification** | **Definition** | **Action Required** |
| --- | --- | --- |
| **True Positive (TP)** | A real attack/malicious behavior confirmed by evidence101010. | **Contain** the threat and **Escalate** to Tier 2/3 or IR111111. |
| **False Positive (FP)** | A false alarm; legitimate activity mistaken for a threat121212. | **Tune** the detection rule or **Whitelist** the activity to reduce noise131313. |
| **Need More Info (NMI)** | Insufficient data to make a conclusion141414. | Collect more logs or confirm with the user/IT department151515. |
| **True Negative (TN)** | System is functioning normally; no alert and no threat1616. | No action; information can be safely ignored1717. |
| **False Negative (FN)** | **Most Dangerous**: A real attack is happening but the system fails to alert18181818. | Improve detection coverage to eliminate "blind spots"191919. |

## 3. SOC Analyst Mindset & Principles

![image.png](Homework%20-%20Day%202%20(05%2001%202026)/image%202.png)

- **Evidence over Emotion:** Never make decisions or guesses based on feelings; rely strictly on evidence.
- **Context is King:** A "High" severity alert without context is weaker than a "Medium" alert with clear context.
- **Maintain Visibility:** Never disable a security system just to stop False Positives; instead, tune the rules.
- **Collaboration:** Do not work in isolation; escalate when uncertain.

## 4. Operational Workflow

The lifecycle of an alert follows an 8-step process :

1. **Behavior:** User logs in, accesses a website, or runs a process.
2. **Log Generation:** Data is created by EDR, Firewalls, or Cloud audits.
3. **Collection & Normalization:** Logs are sent to the **SIEM** and standardized.
4. **Detection:** Logs are mapped against rules; data is enriched (GeoIP, VirusTotal).
5. **Alerting:** An alert is generated with a name, severity, and artifacts.
6. **Triage:** The analyst classifies the alert and builds a timeline.
7. **Decision:** The verdict is reached (TP, FP, or NMI).
8. **Feedback:** Rules are tuned, and playbooks/whitelists are updated to improve future detection.

## 5. Case study

### **SOC-101: Unusual Login - Impossible Travel Detected**

![image.png](Homework%20-%20Day%202%20(05%2001%202026)/image%208.png)

```powershell
Alert Name: SOC-101: Unusual Login - Impossible Travel Detected
Source: Office network in Hanoi
Destination: IP 94.159.113.15 (Foreign country)
Timestamp: Aug 07 2025, 9:13 AM
Action: Login attempt successful (detected and alerted by the SIEM/Security system)
Severity: High
Artifact: * User account: thuy.le@finova.one
Source IP: 94.159.113.15
Context: The user logged into the system from a foreign IP address only 10 minutes after a previous login from the Hanoi office. Because the physical distance between these two locations cannot be traveled in 10 minutes, the activity is flagged as "Impossible Travel," suggesting potential credential compromise or unauthorized VPN usage.
```

### Recommendation & Workflow for SOC-101 alert

**Auto-check IP reputation:**

- Immediately run the foreign IP (**94.159.113.15**) through tools like **VirusTotal**, **AbuseIPDB**, or **Cisco Talos**.
- **What to look for:** Is the IP flagged as a known Malicious Exit Node, a Proxy, or a common VPN provider (e.g., NordVPN, ExpressVPN)?

**Observe and verify user pattern:**

- Check the user's login history for the last 30 days. Does `thuy.le` frequently use a VPN or travel to this specific country?
- Check for other "High" severity alerts for this user (e.g., failed logins followed by success, or unusual file access).

**Direct Verification (Out-of-Band):**

- Contact the user via a trusted channel (Teams/Slack or phone) to ask: *"Are you currently traveling or using a VPN for work?"*

**Immediate Containment (If suspicious):**

- If the IP is flagged as malicious or the user denies the activity:
- **Reset the password** immediately.
- **Revoke all active sessions** (Sign out of all devices).
- Enforce/re-verify **Multi-Factor Authentication (MFA)**.

**Review Post-Login Activity:**

- Look at what the user did *after* the suspicious login. Did they download large amounts of data, change security settings, or create new accounts?
