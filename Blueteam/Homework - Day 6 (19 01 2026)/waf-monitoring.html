WAF Engineering, SOC Detection & Log Analysis

From deploying and tuning Web Application Firewalls to engineering detection rules that catch attackers probing for bypasses — plus command injection hardening, PowerShell forensics, and how the five-layer security stack fits together in a mature SOC.

Core Theory 5 Sections WAF · SIEM · EDR · SOAR · Command Injection · Log Analysis

1. WAF Engineering & Monitoring

The WAF is your first line of telemetry — not just a blocking tool. Understanding how to deploy, tune, and monitor it correctly is foundational to shifting from a reactive to a proactive security posture.

Strategic Deployment

Cloud WAFs are operationally convenient but carry a critical blind spot: if an attacker discovers your Origin IP through DNS history, certificate transparency logs, or other means, they can route requests directly to your server — completely bypassing the WAF.

The Visibility Gap

Cloud WAFs can be fully circumvented if an attacker finds the Origin IP and hits it directly. All your rules and logging are bypassed in a single move.

Defence: IP Whitelisting

Configure the origin server's firewall to only accept inbound traffic from the WAF's published IP ranges. All other sources get dropped at the network level.

Count Mode Before Block

Always run new WAF rules in "Count / Log" mode first. Calculate the False Positive Rate before switching to "Block" — otherwise you risk disrupting legitimate traffic.

Tuning Security Models

Positive Model (Strict) — API Endpoints

Define exactly what is allowed. Use JSON Schema validation on endpoints like /api/v1/login to reject any unexpected keys or field types. Anything outside the schema is blocked by default.

Negative Model (Signature) — Legacy Apps

Define what is not allowed. Essential for legacy applications that cannot be easily refactored. Requires weekly virtual patching as new CVE-specific signatures emerge.

Operational Insight — SQLi: Reactive to Proactive

A common failure is detecting a SQLi bypass only after Threat Intel reports it. The fix is a SIEM correlation rule that catches the bypass in real time:

Rule Logic:
  WAF_Log.payload CONTAINS ("UNION" OR "SELECT")
  AND HTTP_Response.status == "200 OK"
  AND HTTP_Response.body_size > BASELINE_THRESHOLD
 
Action: Trigger HIGH-PRIORITY alert # potential successful data leak

The key insight: a SQL keyword alone is noise. A SQL keyword that returns HTTP 200 with an abnormally large response body signals the payload may have executed and data is leaking out.

2. Detecting WAF Probing & Bypass Attempts

Attackers don't jump straight to exploitation — they probe limits first. Recognising this recon phase early is what separates a good SOC from a great one.

High-Fidelity Detection Indicators

Impedance Mismatch

Monitor URL parameters containing /* and */. These characters are used to "sandwich" payloads between comment delimiters, confusing WAF parsers that process differently from the backend.

Protocol Anomalies

Alert on Transfer-Encoding: chunked combined with high-risk keywords. Treat double-encoded characters like %2527 as high-fidelity attack indicators — legitimate clients almost never produce them.

Payload Padding / Size Evasion

If your WAF has a 10KB inspection limit, create a "Size Limit Exceeded" alert. Attackers prepend junk data to push malicious payloads past the inspection buffer threshold.

Database Behaviour Monitoring

Even if a WAF bypass succeeds, anomalous database behaviour is your second detection opportunity. Instrument your database audit logs with these rules:

Query Volume Spike

Alert when the web-app service account executes a high volume of queries within a short time window — a hallmark of automated data extraction or blind SQLi enumeration.

High-Value Table Access

Alert when the service account touches tables like users, configs, or secrets that standard API logic has no business reason to query directly.

Detection Philosophy — 403 vs 200

A surge of 403 responses means the WAF is working correctly. A 403 followed by a 200 OK on a slightly modified version of the same payload is the smoking gun for a successful bypass. Build a SIEM correlation rule on this exact sequence.

3. Command Injection Mitigation

Instead of playing whack-a-mole with payload signatures, focus on the execution context and process lineage. A web server should never be spawning a shell — if it does, that's your signal regardless of what payload triggered it.

Hardening & Prevention

1

Least Privilege Process Accounts

Run web services under www-data or nobody. Even if command injection succeeds, the attacker inherits minimal filesystem and network permissions.

2

PHP Hardening — Disable Dangerous Functions

In php.ini, add to disable_functions: exec, passthru, shell_exec, system, proc_open, popen. These are virtually never needed in production.

3

Internal Command Mapping

Never pass raw user input as a filename or shell argument. Instead, accept a numeric ID (input=1) and map it server-side to a known-safe value. The user never controls what reaches the shell.

Detection Engineering — PowerShell & SOAR Automation

Manual decoding of obfuscated PowerShell is a bottleneck that inflates MTTR. The goal is to automate the tedious parts so analysts can focus on intent.

Case Study — Automating PowerShell Triage

EDR Process Lineage Rule:
  Parent_Process IN (w3wp.exe, httpd.exe, nginx.exe)
  AND Child_Process IN (powershell.exe, cmd.exe)
 
Production Action: AUTO-TERMINATE / BLOCK # not just alert
 
SOAR Enrichment (on Base64/EncodedCommand detection):
  1. Extract Base64 string from log
  2. Decode: From Base64 → UTF-16LE
  3. Append plaintext to analyst ticket automatically

SOAR automation handles the mechanical decoding; the analyst's job is to classify intent — not to run CyberChef manually on every alert.

Fileless / LOLBin Signatures to Monitor

Downloader Pattern

Watch command lines for Net.WebClient, DownloadString, iwr (Invoke-WebRequest). These pull remote payloads into memory without touching disk.

In-Memory Execution

IEX (Invoke-Expression) executes strings directly — the canonical "download and run" pattern. Alert on any occurrence combined with a URL or decoded payload.

Hidden Window Flag

-WindowStyle Hidden or -W Hidden suppresses the PowerShell console from appearing. Legitimate admin scripts almost never need this in production environments.

Onion Recovery — When You Find EncodedCommand in Logs

EXT

Extract

Pull the Base64 string from the SIEM log field — usually found in CommandLine or ProcessArgs.

DEC

Decode

Use CyberChef: From Base64 → Decode Text (UTF-16LE). PowerShell's -EncodedCommand always uses UTF-16LE encoding.

INT

Classify Intent

Is it a Downloader? (Look for Net.WebClient or iwr.) A Reverse Shell? (IP + port.) Persistence? (Registry keys or Scheduled Tasks.)

4. SOC Log Analysis — Separating Noise from Signal

Raw logs are noise. A SOC analyst's core skill is building mental (and automated) models that filter noise and surface true signals. The "Forensic Onion" approach layers evidence sources to build an accurate picture.

Access Log Triage

What You See in Logs What It Likely Means Action
Surge of 403 responses on similar paths WAF is blocking an automated scanner or attacker probing Monitor — WAF is working. Enrich with GeoIP & ASN.
403 → 200 OK on modified payload Attacker found a WAF bypass — possible successful exploit HIGH PRIORITY — Escalate immediately. Pull PCAP.
POST requests with no body in logs POST data not captured — blind spot in your logging pipeline Enable PCAP or RASP to capture request bodies.
EncodedCommand in Windows Event 4688 Obfuscated PowerShell — almost always malicious in production Base64 decode → classify intent → escalate if downloader/shell.

POST Data Blind Spot

Web server access logs typically capture URL, headers, and status codes — but not POST body content. If your attacker sent a SQLi payload in a POST body and your logs show only a 200, you need PCAP or RASP (Runtime Application Self-Protection) to see what was actually submitted.

5. Security Tool Interoperability

No single tool sees everything. A mature SOC operates across five distinct layers — each catching what the others miss. Understanding which layer is responsible for which detection is what makes alert triage fast and accurate.

Layer Component Blue Team Responsibility
Edge WAF Filter ~90% of internet noise and automated scans before they reach application logic. First line of telemetry.
App RASP / ADR Verify whether a payload actually triggered a vulnerable internal function. Confirms exploitation vs. mere delivery.
Host EDR Catch lateral movement or unauthorised process spawning — e.g. nginx → sh. Process lineage is the gold standard for post-exploitation detection.
Data DB Audit Logs Monitor for exfiltration patterns and unauthorised table access. Catches successful SQLi that the WAF missed.
Network Egress Filtering Block unauthorised outbound connections (e.g. Burp Collaborator callbacks, reverse shells). Prevents C2 establishment and data exfiltration even after initial compromise.

The Layered Defence Mindset

When triaging an alert, always ask: "Which layer caught this, and what does that tell me about the attacker's current stage?" A WAF alert means attempted delivery. An EDR process alert means successful execution. A DB audit alert means successful exploitation. Each layer narrows the blast radius assessment.

Key Takeaways

  • Cloud WAFs must be paired with origin IP whitelisting — otherwise a discovered Origin IP renders the WAF useless.
  • Always run new WAF rules in Count mode first; calculate false positive rate before blocking.
  • A 403 → 200 sequence on similar payloads is the strongest signal for a successful WAF bypass — build a SIEM rule specifically for this pattern.
  • SQLi detection is most reliable when WAF logs are correlated with DB response size, not just keyword matching.
  • Process lineage (web server spawning a shell) is more reliable than payload signatures for command injection detection — set to auto-terminate in production.
  • SOAR should auto-decode Base64 / EncodedCommand strings and attach the plaintext to analyst tickets to reduce MTTR.
  • Classify decoded PowerShell by intent: Downloader, Reverse Shell, or Persistence — each has a different response playbook.
  • POST body content is invisible in standard access logs — use PCAP or RASP to close this visibility gap.
  • Five-layer stack (WAF → RASP → EDR → DB Audit → Egress) — know what each layer sees and what it misses.
